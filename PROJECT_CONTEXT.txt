1. ИСХОДНАЯ ЦЕЛЬ ПОЛЬЗОВАТЕЛЯ
================================================================================
- Собрать интересный портфельный проект на тему мультиагентной автоматизации
  (конвейер документов) для демонстрации работодателю на позицию инженера по
  автоматизации.
- Выбран вариант №2: «офисный конвейер документов» — входящие файлы/письма →
  извлечение сущностей → сверка с CRM/таблицей → шаблон → маршрутизация с
  human-in-the-loop при необходимости. Реализовано: Intake, Extract (эвристики по тексту
  PDF), Match (локальная CRM-сверка по ИНН), Route (правила маршрутизации),
  Template (готовый текст документа + email-отправка), веб
  (цепочка intake → extract → match → route → template/send).
  Не реализовано: OCR, LLM, внешняя CRM/БД, оркестратор workflow/очередь, HITL.

================================================================================
2. АРХИТЕКТУРНЫЕ РЕШЕНИЯ (СОГЛАСОВАННЫЕ В ДИАЛОГЕ)
================================================================================
2.1. Структура репозитория (логика разделения)
  - core/     — общие настройки, схемы данных (Pydantic), без привязки к «личности» агента.
  - tools/    — внешние действия и утилиты: хранение файлов, разбор PDF и т.д.
  - agents/   — подпакет на каждого агента: intake, extract, match, route, template.
  - web/      — только HTTP/UI и вызов уже существующей логики агентов (отдельно,
                чтобы не смешивать доменную логику и интерфейс).
  - orchestration/ (папка задумывалась для оркестратора цепочки) — пока не создана;
    в UI: один POST /api/intake (multipart file) всегда возвращает Intake + Extract в одном JSON;
    опционально отдельно POST /api/extract по job_id.

2.2. Имя и роль первого агента
  - Пакет: agents/intake
  - Функция входа: run_intake(path: Path) -> IngestionResult
  - Смысловое имя: агент приёма (Intake) — НЕ извлекает суммы/ИНН; готовит задачу
    для следующих агентов.

2.3. Второй агент — Extract (извлечение)
  - Пакет: agents/extract
  - Вход: IngestionResult (или только job_id через intake.json) при успешном Intake.
  - run_extract / run_extract_by_job_id -> ExtractionResult; текст только из PDF без OCR;
    растр и «пустой» PDF -> status skipped_ocr. Эвристики: ИНН 10/12 цифр, даты ДД.ММ.ГГГГ,
    суммы с , / . и «руб». Результат пишется в data/jobs/{job_id}/extraction.json.

2.4. Третий агент — Match (CRM-сверка)
  - Пакет: agents/match
  - Вход: ExtractionResult (или только job_id через extraction.json) после успешного Extract.
  - run_match / run_match_by_job_id -> MatchingResult; сверка по кандидатам ИНН с локальным
    реестром CRM (tools/crm_registry.py). Результат пишется в data/jobs/{job_id}/matching.json.

2.5. Четвёртый агент — Route (маршрутизация решения)
  - Пакет: agents/route
  - Вход: extraction.json + matching.json (через job_id).
  - run_route_by_job_id -> RoutingResult; rule-based decision:
    ready_for_template / review_required / rejected / failed.
  - Пишет data/jobs/{job_id}/routing.json.

2.6. Пятый агент — Template (подготовка документа + отправка email)
  - Пакет: agents/template
  - Вход: extraction.json + matching.json + routing.json (через job_id).
  - run_template_by_job_id -> TemplateResult; при route ready_for_template формирует
    data/jobs/{job_id}/prepared_document.txt и manifest template.json.
  - send_template_by_job_id(job_id, recipient_email) -> EmailSendResult; отправка
    prepared_document.txt на email через SMTP (переменные окружения SMTP_*), журнал
    отправки в data/jobs/{job_id}/email_send.json.

3. АГЕНТ INTAKE — ЗАДАЧИ И ПОВЕДЕНИЕ
================================================================================
3.1. Ответственность
  - Принять путь к файлу; проверить существование, расширение, размер.
  - Допустимые расширения: .pdf, .png, .jpg, .jpeg (см. core/config.py).
  - Максимальный размер: 25 МБ.
  - При успехе: сгенерировать job_id (UUID), скопировать файл в
    data/jobs/{job_id}/original.<ext> (см. tools/storage.py).
  - Грубая классификация по ИМЕНИ файла (эвристики): invoice / contract / other
    (см. agents/intake/agent.py, маркеры на русском и английском).
  - PDF: tools/pdf_inspect.py — число страниц; выборка текста с первых N страниц
    (PDF_SAMPLE_PAGES); если символов мало — needs_ocr = true (скан).
  - Растровые изображения: page_count = 1, needs_ocr = true.
  - При ошибке чтения PDF — status rejected, rejection_reason pdf_read_error.
  - При отклонении на ранней стадии файл в data/jobs не копируется (job_id всё равно
    генерируется для трассировки в ответе).

3.2. Модели данных (core/schemas.py)
  - DocumentKind: invoice | contract | other | unknown
  - IngestionStatus: accepted | rejected
  - IngestionResult: job_id, status, original_filename, stored_path, document_kind,
    kind_confidence, needs_ocr, page_count, rejection_reason, notes
  - Модели Extract (ExtractionResult и др.) — см. п. 4.2.
  - Модели Match (MatchingResult и др.) — см. п. 4.6.
  - Модели Route (RoutingResult и др.) — см. п. 4.7.
  - Модели Template/Email (TemplateResult, EmailSendResult и др.) — см. п. 4.8.

3.3. CLI
  - Запуск из корня проекта: python -m agents.intake <путь_к_файлу>
  - Вывод: JSON в stdout; код выхода 0 при accepted, 1 при rejected.

3.4. Манифест intake.json
  - При status accepted в каталоге data/jobs/{job_id}/ создаётся файл intake.json
    (полный JSON IngestionResult) для агента Extract и API /api/extract.

================================================================================
4. АГЕНТ EXTRACT — ИЗВЛЕЧЕНИЕ СУЩНОСТЕЙ
================================================================================
4.1. Назначение
  - Второй шаг пайплайна: по сохранённому файлу и данным Intake извлечь текст из PDF
    (pypdf, лимиты MAX_EXTRACT_PAGES / MAX_EXTRACT_CHARS в core/config.py) и найти
    кандидаты сущностей regex-ами (tools/entity_patterns.py): ИНН 10/12 цифр, даты
    ДД.ММ.ГГГГ, денежные строки с , / . и опционально «руб» / ₽.
  - Растровые изображения: без Tesseract — сразу status skipped_ocr.
  - Если символов текста < MIN_TEXT_CHARS_FOR_EXTRACTION — skipped_ocr (скан/пустой слой).

4.2. Модели (core/schemas.py)
  - ExtractionStatus: ok | skipped_ocr | failed
  - ExtractedEntities: inn_candidates, date_candidates, money_candidates (списки строк)
  - ExtractionResult: job_id, status, document_kind, text_char_count, text_preview,
    entities, rejection_reason, notes

4.3. Файлы кода
  - agents/extract/agent.py — run_extract(IngestionResult), run_extract_by_job_id(job_id);
    при успехе ok пишет data/jobs/{job_id}/extraction.json.
  - tools/pdf_text.py — extract_pdf_plain_text
  - tools/job_manifest.py — load_intake_result(job_id) из intake.json

4.4. CLI
  - python -m agents.extract <job_id> — читает intake.json в каталоге job; stdout JSON;
    код 0 только при status ok.

4.5. Проверка связки Intake + Extract (скрипт)
  - scripts/test_two_agents.py — из корня проекта: python scripts/test_two_agents.py
    Пишет минимальный PDF в data/_test_run/schet_demo.pdf, вызывает run_intake и
    run_extract_by_job_id; код выхода 0 при успехе. Без доп. pip-пакетов.
  - Для ручного теста с «богатым» набором сущностей используется
    data/_test_run/schet_rich_demo.pdf (несколько INN, даты и суммы в разных форматах).

4.6. Третий агент Match (CRM-сверка по ИНН)
  - Модели (core/schemas.py):
    - MatchStatus: ok | no_match | skipped | failed
    - CRMMatchItem: inn, company_name, account_manager, segment, source
    - MatchingResult: job_id, status, checked_inn, matches, rejection_reason, notes
  - Логика:
    - Если Extract не ok -> status skipped (rejection_reason extract_not_ok)
    - Если entities отсутствуют -> status skipped (entities_missing)
    - По списку INN-кандидатов после нормализации (только 10/12 цифр) ищет совпадения в
      локальном CRM-реестре tools/crm_registry.py
    - При совпадениях: status ok; без совпадений: status no_match
    - Результат пишется в data/jobs/{job_id}/matching.json
  - CLI:
    - python -m agents.match <job_id> — читает extraction.json и запускает сверку;
      код 0 только при status ok.
  - Проверка цепочки Intake + Extract + Match:
    - scripts/test_three_agents.py — из корня проекта: python scripts/test_three_agents.py
      (минимальный PDF -> intake -> extract -> match).

4.7. Четвёртый агент Route (финальная маршрутизация)
  - Модели (core/schemas.py):
    - RouteStatus: ready_for_template | review_required | rejected | failed
    - RoutingResult: job_id, status, queue, priority, reasons, next_actions, rejection_reason, notes
  - Логика (agents/route/agent.py):
    - Если Extract не ok -> review_required, queue=ocr_review, priority=high
    - Если Match ok -> ready_for_template, queue=templating
    - Если Match no_match -> review_required, queue=crm_enrichment
    - Если Match skipped -> review_required, queue=doc_control
    - Иначе -> failed
  - Выход:
    - data/jobs/{job_id}/routing.json
  - CLI:
    - python -m agents.route <job_id> — читает extraction.json + matching.json и строит routing.
  - Проверка цепочки Intake + Extract + Match + Route:
    - scripts/test_four_agents.py — из корня проекта: python scripts/test_four_agents.py.

4.8. Пятый агент Template (готовый документ и email-отправка)
  - Модели (core/schemas.py):
    - TemplateStatus: ready | skipped | failed
    - TemplateResult: job_id, status, template_path, rejection_reason, notes
    - EmailSendStatus: sent | failed
    - EmailSendResult: job_id, status, recipient_email, subject, rejection_reason, notes
  - Логика (agents/template/agent.py):
    - Если route.status != ready_for_template -> template status skipped
    - Если match.status != ok -> template status skipped
    - Иначе формирует prepared_document.txt (контрагент, ИНН, даты/суммы, route-решение)
      и пишет template.json.
    - Отправка email через tools/email_sender.py (SMTP_HOST, SMTP_PORT, SMTP_USER,
      SMTP_PASSWORD, SMTP_FROM, SMTP_USE_TLS), результат в email_send.json.
  - CLI:
    - python -m agents.template <job_id> — генерация prepared_document.txt.
  - Проверка цепочки Intake + Extract + Match + Route + Template:
    - scripts/test_five_agents.py — из корня проекта: python scripts/test_five_agents.py.

5. ВЕБ-ИНТЕРФЕЙС (ПАПКА web/)
================================================================================
5.1. Технологии
  - FastAPI, uvicorn, python-multipart (загрузка файлов).

5.2. Файлы
  - web/app.py — приложение FastAPI:
      GET  /        — отдаёт web/static/index.html (форма загрузки), Cache-Control: no-store (свежий JS).
      POST /api/intake — multipart: только file. После Intake при status accepted вызывается run_extract(ingestion)
                         в памяти (без повторного чтения intake.json); ответ всегда {"intake": ..., "extract": ... | null}.
                         При rejected extract = null. Исключения внутри Extract сериализуются как extract с status failed
                         (rejection_reason extract_exception), а не обрывают ответ 500.
      POST /api/extract и POST /api/extract/ — JSON {"job_id": "..."}; для CLI/интеграций;
                         404 с телом {error, detail} если нет job или манифеста.
      POST /api/match и POST /api/match/ — JSON {"job_id": "..."}; запускает CRM-сверку по
                         extraction.json, 404 если job или extraction.json отсутствуют.
      POST /api/route и POST /api/route/ — JSON {"job_id": "..."}; строит routing по
                         extraction.json + matching.json, 404 если нет манифестов.
      POST /api/template и POST /api/template/ — JSON {"job_id": "..."}; формирует
                         prepared_document.txt и template.json.
      POST /api/send-template и POST /api/send-template/ — JSON {"job_id", "recipient_email"};
                         отправляет prepared_document.txt по SMTP.
    Импорты: agents.intake, agents.extract, agents.match, agents.route, agents.template
             (корень проекта в PYTHONPATH при cwd = корень «Агент»).
  - web/static/index.html — UI: POST /api/intake (только файл в FormData), пять блоков JSON
                         (intake + extract + match + route + template). Если Intake accepted, а поля extract нет
                         или оно null — запасной POST /api/extract по job_id (совместимость со
                         старым воркером uvicorn, отдававшим плоский JSON Intake). При extract ok
                         вызывается POST /api/match по job_id; затем POST /api/route по job_id и
                         POST /api/template по job_id; после этого доступно поле email и кнопка
                         отправки через POST /api/send-template.
    Запросы на window.location.origin; страница с http://127.0.0.1:8000/.

5.3. CORS
  - В app.py включён CORSMiddleware allow_origins=["*"] (удобно для локальной разработки).

6. ЗАПУСК СЕРВЕРА И ОКОННАЯ МОДЕЛЬ WINDOWS
================================================================================
6.1. web/run_web.py
  - Определяет ROOT = родитель каталога web (т.е. корень проекта «Агент»).
  - Подпроцесс: uvicorn … --reload --reload-dir ROOT --reload-include web/reload.trigger
    (перезапуск при изменении .py в проекте и при обновлении mtime файла web/reload.trigger).
  - web/app.py: middleware на GET / — при «настоящем» обновлении вкладки (F5: Sec-Fetch-Mode:
    navigate + Referer + не urllib) вызывается touch на web/reload.trigger → uvicorn перезапускает
    воркер. Первый заход без Referer и проверка из run_web (Python-urllib) не трогают файл —
    без лишнего цикла перезапусков.
  - Файл web/reload.trigger в .gitignore.
  - cwd подпроцесса = ROOT (иначе ломаются импорты и пути data/).
  - Ожидание готовности: опрос http://127.0.0.1:8000/ через urllib; затем webbrowser.open.
  - Windows: subprocess.CREATE_NO_WINDOW — чтобы дочерний uvicorn не открывал консоль.
  - Сообщения print в консоль только если stdout.isatty() (удобно при ручном запуске
    из терминала; при pythonw — тихо).

6.2. web/start_web.vbs
  - Двойной щелчок БЕЗ чёрного окна cmd.
  - Запускает .venv\Scripts\pythonw.exe web\run_web.py если venv есть, иначе pyw -3.
  - WorkingDirectory (через WScript.Shell.CurrentDirectory) = корень проекта.
  - sh.Run cmdLine, 0, False — скрытое окно, не ждать завершения (скрипт выходит,
    процесс pythonw с сервером остаётся в фоне).

6.3. Удалённые / заменённые артефакты запуска
  - Ранее в корне были run_web.py и start_web.bat — перенесены в web/, затем bat удалён
    из-за невозможности убрать всплывающее окно cmd полностью.
  - web/start_web.bat (с cd .. и pause) удалён; web/start_web_debug.bat создавали и
    затем удалили по запросу пользователя («зачем он нам»).
  - Итог: основной запуск для пользователя — start_web.vbs; отладка вручную:
    .\.venv\Scripts\python web\run_web.py

6.4. Остановка «тихого» сервера
  - Нет окна — нет Ctrl+C. Остановка: диспетчер задач → завершить pythonw.exe
    (или соответствующий процесс). В будущем можно добавить кнопку «Стоп» в UI или
    endpoint — пока не сделано.
